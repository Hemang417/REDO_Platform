REDO Platform

AI-Powered Deal Origination
for Maharashtra Real Estate

REDO is a six-module Python data pipeline that monitors every MAHARERA-registered project across Maharashtra, scores each one as an investment opportunity, and uses an AI model to write a structured investment memo — producing a deal origination report ready for your investment committee.

It runs entirely on your machine. No cloud database, no web server. Each module writes plain JSON and CSV files that the next module reads. You can run the full pipeline end-to-end in one sitting, or run each module individually when data needs refreshing.

ⓘ
What REDO does NOT do

It never invents facts. Every investment thesis the AI writes is grounded exclusively in structured MAHARERA data. If a field is missing, the memo says so. No external market data, no assumptions.

How the Pipeline Works

Data flows through six modules in sequence. Each module is a standalone Python script that reads from the previous module's output folder and writes its own output folder. Nothing is stored in memory between runs — everything is a file.

Module 1

Collector — Scraping MAHARERA

1
MAHARERA Collector
scripts/run_collector.py  ·  src/scraper/
Scrapes every registered project from the MAHARERA portal and saves raw data. MAHARERA has no public API — the portal serves list pages as server-rendered HTML and detail pages as a React app behind a CAPTCHA. The collector handles both tiers.
Input
MAHARERA portal (live web)
config/settings.yaml
Output
output/raw/maharera_projects_*.json
output/raw/maharera_projects_*.csv

How It Works

MAHARERA has two separate domains with different behaviours:

DomainWhat it servesHow we access it
maharera.maharashtra.gov.inProject list pages (server HTML)BeautifulSoup + requests
maharerait.maharashtra.gov.inProject detail pages (React SPA + API)JWT token via POST API

The first time you run the collector, Playwright opens a real Chrome window so you can solve the CAPTCHA once manually. After you solve it, the JSESSIONID and JWT token are saved locally. All subsequent requests use those saved tokens — Chrome is never opened again unless the token expires.

⚠
The MAHARERA portal has ~48,700 projects across ~4,873 pages.

A full scrape at 1.5 s/request takes roughly 22 hours. Always test with --end-page 2 first (20 projects, ~30 seconds). Use --start-page to resume from a checkpoint.

Run Commands

powershell
# Test run — first 2 pages only (20 projects)
python scripts/run_collector.py --end-page 2

# Dry run — scrapes list pages only, no detail fetches
python scripts/run_collector.py --end-page 2 --dry-run

# Resume from page 500 after a previous interrupted run
python scripts/run_collector.py --start-page 500

# Full scrape (all pages — long running)
python scripts/run_collector.py

Converts raw strings from the collector into properly-typed Python values. A project with "31/12/2025" as a date string becomes a datetime.date. Delay days are computed. Location names are normalised. Records that fail validation are dropped with a warning.
Input
output/raw/maharera_projects_*.json
Output
output/clean/maharera_clean_*.json
output/clean/maharera_clean_*.csv

What it does to each record

TransformExample
Date strings → date objects"31/12/2025" → 2025-12-31
Number strings → float/int"67.5" → 67.5
Status flags → bool"Lapsed" → is_lapsed=True
Delay computationdelay_days = today − proposed_completion_date (positive = overdue)

Module 3

Scorer — Opportunity Score Engine

3
MAHARERA Scorer
scripts/run_scorer.py  ·  src/scorer/
Scores each cleaned project on a 0–100 scale using five weighted factors defined in config/scoring_rules.yaml. The score reflects how attractive a project is as an AIF investment target — the "sweet spot" is a project that is 40–85% built, moderately delayed, legally active, and in a tier-1 district.
Input
output/clean/maharera_clean_*.json
config/scoring_rules.yaml
Output
output/scored/maharera_scored_*.json
output/scored/maharera_scored_*.csv

Scoring Factors

FactorWeightWhat it measures
Construction Progress30%40–85% complete = ideal (past risk phase, exit visible). 100% = no funding needed.
Delay Severity25%6 months–2 years delayed = likely funding gap. >3 years = probable distress.
Extension History15%1–2 extensions = classic AIF candidate. 0 = no distress signal. 3+ = risky.
Project Viability20%Legal status. Acts as a multiplier — lapsed or deregistered always scores 0.
Location10%Tier 1 (Mumbai, Thane, Pune) = full score. Tier 2 = 65%. Other = 35%.
ⓘ
Viability is a multiplier, not an additive weight

A lapsed project could theoretically score 80/100 on all other factors. With viability as a multiplier, it scores 0. A project with no legal standing cannot receive AIF investment — so the scoring model enforces this as a hard gate, not just a penalty.

Run Command

powershell
# Auto-finds latest clean file
python scripts/run_scorer.py

# Show top 20 by score, minimum score 30
python scripts/run_scorer.py --top 20 --min-score 30

# Explicit input
python scripts/run_scorer.py --input output/clean/maharera_clean_20260101.json

All scoring thresholds and weights live in config/scoring_rules.yaml. Change them there — never in code.

Output

MetricWeightWhat it measures

Module 5

AI Analyst — Investment Memo Generation

5
MAHARERA Analyst

For each scored project above a threshold, builds a structured project brief from the data, sends it to an AI model, and receives a forced-schema investment memo back. The AI cannot free-text — it must call a function with specific fields, guaranteeing parseable structured output every time.

What Each Memo Contains

FieldDescription
recommended_actionFLAG_FOR_REVIEW / MONITOR / PASS
opportunity_thesis2–3 sentences citing specific metrics. No invented facts.
risk_flagsList of risks, each citing a specific data point
data_gapsFields that were null/missing in the brief
confidence_score0.0–1.0 reflecting data completeness

Cost Safeguards

Two built-in safeguards prevent runaway API spend:

  • Score threshold — set min_score_for_memo in ai_config.yaml (or --min-score flag) to skip low-scoring projects
  • File cache — memos are cached by project ID + score hash. Re-running the analyst never re-bills the same project unless its score changed

Run Commands

$env:GROQ_API_KEY = "gsk_..."
python scripts/run_analyst.py --provider groq

# Only memo projects scoring above 40, show top 10
python scripts/run_analyst.py --provider groq --min-score 40 --top 10

# Explicit files
python scripts/run_analyst.py --provider groq `
  --scored output/scored/maharera_scored_20260101.json `
  --intelligence output/intelligence/developer_profiles_20260101.json

Config Files

All business rules live in YAML. The code never hardcodes thresholds, weights, or URLs.

FileControls

config/scoring_rules.yamlAll 5 scoring factor weights and thresholds; developer track record scoring weights and thresholds

⚠
Never put API keys in YAML files

API keys go in environment variables only ($env:GROQ_API_KEY, $env:ANTHROPIC_API_KEY). The config files are version-controlled; environment variables are not.

Full Pipeline Run — End to End

Here is the exact sequence to run the entire platform from scratch on a fresh dataset:

powershell — complete pipeline
# ── Step 0: Set your API key ────────────────────────────────
$env:GROQ_API_KEY = "gsk_..."

# ── Step 1: Collect (Chrome opens for CAPTCHA — solve it once)
#           For a full run, remove --end-page (takes ~22 hrs)
python scripts/run_collector.py --end-page 5

# ── Step 2: Clean ───────────────────────────────────────────
python scripts/run_cleaner.py

# ── Step 3: Score ───────────────────────────────────────────
python scripts/run_scorer.py

# ── Step 4: Developer Intelligence ─────────────────────────
python scripts/run_intelligence.py

# ── Step 5: AI Analyst (Groq — free tier) ───────────────────
python scripts/run_analyst.py --provider groq

# ── Step 6: Generate Report ─────────────────────────────────
python scripts/run_reporter.py

# ── Open the report ─────────────────────────────────────────
# Find the file at: output/reports/deal_origination_*.html
# Open in Chrome → File → Print → Save as PDF to share
✓
Each step is independent and resumable

If Module 3 (Scorer) fails, fix the issue and re-run only run_scorer.py. You do not need to re-scrape or re-clean. Every module auto-discovers the latest output from the previous module.
